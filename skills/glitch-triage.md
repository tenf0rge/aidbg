# skill: glitch-triage（グリッチ検知SVA：本物か見かけか）

## いつ効くか
グリッチ検知SVA（チェッカ）が**発火**したとき。発火＝「グリッチを検知した」の意味。
問うべきは「それが**実回路の本物のグリッチ（DESIGN）**か、**シミュレーション上の
見かけ（VERIFICATION-ENV）**か」。

## 背景知識
- グリッチ検知SVAかどうかは**名前やメッセージから自分で推定**する（外部レジストリに
  依存しない）。`grep-source` でそのアサーションの定義を読み、意図を判断する。
- 本物のグリッチ：対象ネットの近傍時刻に**物理的原因**がある——競合(X)、ヒゲ、
  伝搬遅延による短パルスなど。波形に裏付けが取れる。
- 見かけのグリッチ：物理的原因が無いのに発火する。典型は **0遅延レース / デルタ
  サイクルの評価順**、TB/モデルのドライブ競合、サンプリングのタイミング。これは
  **検証環境側**の問題。

## 手順
1. `grep-log` で発火したアサーション名・時刻 T・対象ネット S を取得。
2. `grep-source --pattern "<assertion_name>"` で定義を読み、グリッチ検知の意図か確認。
3. `query --signal S`（全変化点）と `--time T` 近傍で、**T の直前に物理的原因があるか**
   を見る。X への遷移、競合、極端に短いパルスがあれば物理由来の手掛かり。
4. 物理原因が tranif 競合なら `tranif-contention` の手順に接続（DESIGN 確定が早い）。
5. 近傍に物理原因が無く、変化が同一時刻の評価順に見える → sim-artifact を疑う。

## Design / Verification-env の判定
- 近傍に物理的原因（競合/X/真のヒゲ）あり → **DESIGN**（本物のグリッチ）。
- 近傍に物理的原因が無く 0遅延レース/評価順が疑わしい → **VERIFICATION-ENV**
  （見かけのグリッチ。TB/モデル/サンプリングを是正すべき）。

## 例
- samples/fixture: AOUT のグリッチSVA発火。近傍で AOUT=X、原因は SEL0/SEL1 同時導通
  （tranif 競合）。物理原因あり → **DESIGN**。`tranif-contention` に接続。
- samples/scenario_tb: 同種のSVAが発火するが、波形上 AOUT に競合も X も無く、発火時刻が
  クロックエッジと同一デルタの評価順に依存 → 物理原因なし → **VERIFICATION-ENV**
  （sim-artifact。TB のドライブ/サンプリング順を是正）。
